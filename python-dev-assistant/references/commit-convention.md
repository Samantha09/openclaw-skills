# 提交规范

## 提交信息格式

```
<类型>[可选 范围]: <描述>

[可选 正文]

[可选 页脚]
```

## 类型

| 类型 | 说明 |
|------|------|
| **feat** | 新功能 |
| **fix** | Bug 修复 |
| **docs** | 仅文档变更 |
| **style** | 不影响代码含义的变更（格式、分号等）|
| **refactor** | 既不修复 Bug 也不添加功能的代码变更 |
| **perf** | 性能优化 |
| **test** | 添加或修正测试 |
| **chore** | 构建过程或辅助工具的变更 |

## 示例

### 功能提交
```
feat(dispatcher): 添加 action 自动发现和注册

从指定路径扫描 Python 模块，自动加载带 @action 装饰器的函数和类。
```

### Bug 修复
```
fix(executor): 修复线程池 shutdown 后仍被引用的问题

shutdown() 现在会清空 _executors 字典，防止 get() 返回已关闭的线程池。

修复 #12
```

### 破坏性变更
```
feat(action)!: action 装饰器参数改为关键字参数

BREAKING CHANGE: @action 不再支持位置参数，必须用关键字参数。
迁移: @action("name") → @action(name="name")
```

## 最佳实践

1. 使用祈使语气（"添加功能" 而非 "添加了功能"）
2. 首字母不大写
3. 描述保持在 72 字符以内
4. 正文说明"为什么"而非"做了什么"（what 看 diff）
5. 每个 commit 只做一件事
